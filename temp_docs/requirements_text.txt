1 — PROJECT OVERVIEW
What is it?
Webtoon Reader PWA is a personal, self-hosted web application that aggregates webtoons and manga
chapters from multiple external websites into a single, unified reading experience. It is inspired by
Comick.io and designed to be installed as a native-like app on mobile devices via the Progressive Web
App standard.
Goals
• Aggregate content from any website by configuring CSS selectors — no hardcoded scrapers.
• Provide a Comick-quality reading experience: smooth scrolling, progress tracking, bookmarks.
• Work offline for previously read chapters via service worker caching.
• Be fully installable on Android and iOS as a home screen app.
• Store all user data locally — no accounts, no cloud, complete privacy.
• Run on a local Node.js backend on the user's machine, accessible from their phone via Wi-Fi.
Out of Scope
• No user authentication or multi-user support.
• No cloud sync (all data stays on device / local network).
• No content hosting — the app only scrapes and proxies existing public websites.
• No monetisation features.

2 — SYSTEM ARCHITECTURE
The system is split into two parts: a lightweight Node.js backend that runs on the user's computer and a
React PWA frontend installed on their phone. Communication happens over the local Wi-Fi network.
Layer
Component
Responsibility
Frontend
React + Vite PWA
UI, reader, bookmarks, offline caching via service worker
Backend API
Node.js + Express
Scraping, caching, image proxy, source management
Scraping Engine
Cheerio + Puppeteer
Parse static HTML sites and JS-rendered sites
Local Storage
IndexedDB
Reading progress, bookmarks, chapter cache, settings
Image Proxy
Express route
Bypass CORS restrictions on source site images
Cache
NodeCache (server)
TTL-based caching of scrape results (5 min default)
Data Flow
1. User opens the PWA on their phone (installed as home screen app).
2. PWA makes API requests to the Node.js backend running on the same Wi-Fi network (e.g.
http://192.168.1.5:3001).
3. Backend checks its in-memory cache. On miss, it fetches the source website HTML.
4. Cheerio or Puppeteer parses the HTML using the stored CSS selectors for that source.
5. Parsed data is returned to the PWA as JSON and rendered in the UI.
6. Chapter images are loaded through the image proxy endpoint to bypass CORS.
7. Reading progress and bookmarks are written to IndexedDB on the device.

6 — SOURCE MANAGEMENT SYSTEM
The most technically unique part of the app. Instead of hardcoded scrapers, each source is fully
configurable via CSS selectors stored in a local JSON file on the backend.
How Sources Work
• Each source has a unique ID, name, base URL, render type, and a set of CSS selectors.
• Selectors are organized into 4 groups: Search, Title Page, Chapter List, and Reader Pages.
• The scraper uses whichever selector group is needed for the current API endpoint.
• Cheerio handles static HTML. Puppeteer is used when JavaScript rendering is required.
• Sources can be enabled or disabled without deletion.
• A per-source scrape delay can be configured to avoid rate limiting.
Selector Groups
Group
Selectors Defined
Search
urlTemplate, item container, title text, title link, cover image, coverAttr,
latestChapter, rating
Title Page
title name, cover image, coverAttr, description, genre tags, status, author
Chapter List
chapter item container, chapter number, optional chapter title, link href, release
date
Reader Pages
image element selector, image attribute (src / data-src / data-lazy)

9 — PWA REQUIREMENTS
The app must meet the full PWA criteria to be installable on both Android and iOS as a native-like home
screen application.
• Web App Manifest (manifest.json) with app name, short name, icons (192px + 512px), theme color,
background color, and display: standalone.
• Service Worker registered via Workbox (vite-plugin-pwa) — handles install, activate, and fetch events.
• HTTPS required for production (or localhost for development).
• Installable on Android Chrome and iOS Safari (Add to Home Screen).
• Works in both landscape and portrait orientation.
• Standalone display mode — browser UI chrome hidden when launched from home screen.
• Offline fallback page shown when network is unavailable and no cache exists.
• Background sync for reading progress (future — requires Push API permission).
• App icon and splash screen configured per platform.
• Meta viewport tag: width=device-width, initial-scale=1, viewport-fit=cover (for notched phones).

Local status overrides (enabled/disabled)
Backend Storage (Server)
• sources.json — array of source configs with selectors, stored as a local JSON file.
• NodeCache — in-memory cache for scrape results, cleared on server restart.
• No database required — the backend is stateless beyond sources.json.
Export / Import Format
Users can export all their data as a single .json file containing:
• All bookmarks and shelf assignments
• All reading progress records
• Full reading history
• Local ratings
• Settings preferences
The same file can be re-imported to restore state on a new device or after a reset.
